Fix the internal consistency of the topic system so that topic counts always exactly match the underlying articles. If a topic says it has 7 related articles or sources, then when I click on that topic, I must be able to see all 7 corresponding items. There must never be a mismatch between the topic count shown on the dashboard and the number of actual articles displayed in the topic view.

Make the topic counting logic deterministic and transparent. The count shown for each topic must be based on the exact number of currently stored and visible articles that are tagged to that topic within the active time window. Do not count hidden, filtered out, duplicate, inactive, or clustered-away items unless those same items are also visible when I open the topic. The topic count and the topic detail page must always use the same dataset and the same filtering rules.

Also improve topic clustering and display so that if multiple sources are grouped into one story cluster, the UI clearly explains that. For example, if a topic has 7 source items but only 3 story clusters, show that distinction clearly instead of making it look inconsistent. If the dashboard says 7, the detail view must either show 7 individual source items or explicitly show 3 grouped stories confirmed by 7 sources. It must never look like the number is wrong.
